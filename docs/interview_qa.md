# 面试问答准备

## 1. 你这个项目解决了什么问题？

解决的是业务转化链路不清楚、新策略是否有效无法判断的问题。  
我通过行为日志搭建转化漏斗，先定位用户在哪个环节流失最多，再通过 A/B 测试判断“领券提醒 + 结算页简化”策略是否真的提升支付转化率。

## 2. 为什么要设计四张表？

因为真实业务中用户信息、行为日志、订单和实验分组通常不是放在一张表里的。

- dim_user：记录用户来源、设备、城市层级等静态属性；
- fact_event：记录用户行为事件，是漏斗分析的核心；
- fact_order：记录订单和金额，用于 GMV、ARPU；
- ab_assignment：记录用户进入哪个实验组，避免后续分析时分组混乱。

## 3. 为什么不能只看 treatment 转化率比 control 高？

因为差异可能来自随机波动。  
如果样本量不够，或者两组用户本身有差异，单看转化率容易误判。所以需要做显著性检验，判断差异是否足够大，大到不太可能只是随机造成。

## 4. 你用的是什么检验？

我用的是双样本比例 Z 检验。  
因为这里比较的是两个独立样本的支付转化率，转化结果是 0/1 变量，样本量也比较大，所以可以用比例 Z 检验。

## 5. p-value 怎么解释？

p-value 表示在“新策略没有真实提升”的前提下，观察到当前差异或更极端差异的概率。  
如果 p-value < 0.05，通常认为差异具有统计显著性，可以考虑继续灰度放量。  
如果 p-value >= 0.05，说明当前证据不足，不能直接说策略有效。

## 6. 这个项目最容易被追问哪里？

最容易被追问三个地方：

1. 表结构为什么这么设计；
2. SQL 中去重用户数为什么用 COUNT DISTINCT；
3. A/B 测试为什么要看 p-value，而不是只看转化率。

这三个问题必须会解释。
