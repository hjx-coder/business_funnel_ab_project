# 基于 SQL + Python 的业务转化漏斗分析与策略实验评估项目

## 1. 项目定位

这是一个面向数据分析 / AI 数据分析 / 数据运营分析岗位的求职项目。

项目不是单纯做可视化大屏，而是围绕“用户从访问到支付的转化链路”完成一套完整分析：

1. 数据表设计
2. SQL 指标取数
3. Python 数据分析
4. A/B 测试显著性检验
5. Streamlit 看板展示
6. 业务结论与策略建议

## 2. 业务背景

某线上业务上线了“领券提醒 + 结算页简化”策略，希望提升用户支付转化率。  
业务方不能只看 treatment 组转化率是否更高，还需要判断：

- 用户在漏斗哪一步流失最多；
- 哪些渠道、设备、城市层级的转化表现更差；
- 新策略相对原流程是否有统计显著提升；
- 是否值得继续灰度放量。

## 3. 数据说明

本项目使用模拟业务数据，字段按真实业务日志习惯设计，便于本地复现和面试讲解。

| 表名 | 含义 |
|---|---|
| dim_user | 用户维表 |
| fact_event | 用户行为事件事实表 |
| fact_order | 订单事实表 |
| ab_assignment | A/B 实验分组表 |

数据规模：

- 用户数：5,000
- 行为事件数：22,643
- 订单数：960
- 实验分组数：5,000

## 4. 核心指标

| 指标 | 解释 |
|---|---|
| 首页访问人数 | visit_home 去重用户数 |
| 商品浏览人数 | view_product 去重用户数 |
| 加购人数 | add_cart 去重用户数 |
| 提交订单人数 | submit_order 去重用户数 |
| 支付成功人数 | pay_success 去重用户数 |
| 分步转化率 | 当前步骤人数 / 上一步人数 |
| 总转化率 | 当前步骤人数 / 首页访问人数 |
| ARPU | GMV / 实验组用户数 |
| Lift | treatment 转化率相对 control 的提升比例 |
| p-value | 判断实验组转化提升是否可能由随机波动导致 |

## 7. 运行方式

### 7.1 安装依赖

```bash
pip install -r requirements.txt
```

### 7.2 重新生成数据

```bash
python src/01_generate_data.py
```

### 7.3 建立数据库

```bash
python src/02_build_database.py
```

### 7.4 执行分析

```bash
python src/03_analysis.py
python src/04_ab_test.py
```

### 7.5 启动看板

```bash
streamlit run src/app.py
```

## 8. 面试时的项目一句话

我做的是一个用户转化漏斗与 A/B 实验评估项目，先把用户行为日志拆成用户维表、行为事实表、订单事实表和实验分组表，再用 SQL 计算从访问、浏览、加购、提交订单到支付成功的分步转化率，用 Python 对不同渠道和实验组做分析，并通过双样本比例 Z 检验判断新策略是否真实提升支付转化率，最后用 Streamlit 做成可复现看板。

## 9. 注意

这个项目可以写进简历，但不能说成真实公司线上项目。正确说法是：

## 10. AI 自动分析报告模块

运行方式：

```bash
python src/05_ai_report.py
```

如果配置了大模型 API，会调用模型生成报告；如果没有配置 API Key，会自动使用规则模板生成报告，保证项目本地可运行。
